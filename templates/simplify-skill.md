# SimplifyCode Universal Skill (Captain OS Core)

Этот документ определяет универсальные правила рефакторинга и садоводства кода (Code Gardening) для ЛЛМ-моделей (Codex, Claude, Gemini) в рантайме Captain OS.

---

## 🛡️ Золотые правила безопасности и стабильности

1. **Сохранение внешнего поведения (Functional Integrity)**:
   * **Запрещено** изменять публичные экспорты, сигнатуры функций, типы аргументов и возвращаемые значения. Рефакторинг должен быть на 100% прозрачен для внешней системы.
2. **Вето на логику безопасности (Security Gate Protection)**:
   * **Категорически запрещено** упрощать, сокращать или модифицировать логику авторизации, проверок прав доступа, платежных шлюзов или криптографических процедур (файлы вида `*Gate`, `*auth`, `*security`, `RLS` политики).
3. **Строгая защита старых файлов в Splash Radius (Conscious Agreement)**:
   * **Сплэш-радиус задачи может затрагивать старые стабильные файлы**, но агент **не имеет права** автоматически модифицировать или упрощать их без явного согласия оператора.
   * Любые предложения по упрощению существующего (старого) кода должны запрашиваться через интерактивный консольный диалог и подтверждаться разработчиком вручную.
4. **Свобода оптимизации новых файлов**:
   * Для всех **новых файлов**, созданных с нуля в рамках текущей задачи, агент имеет полное право применять авто-упрощение и Code Gardening без запроса согласия, доводя их до идеальной flat-структуры и безупречной лаконичности.
5. **Изначальная простота и минимализм (KISS from the Start)**:
   * Агенты обязаны писать максимально простой и компактный код **изначально**, а не рассчитывать на последующую фазу упрощения.
   * Стремитесь изменять минимум строк, добавлять минимум функций и классов, полностью опираясь на существующий RAG-контекст и узкий splash-радиус. Избегайте преждевременного усложнения и избыточных абстракций.

---

## 🪚 Принципы рефакторинга читаемости (Readability Tenets)

### 1. Уплощение структуры (Flattening Nesting)
Избегайте «пирамид» из вложенных конструкций `if-else`. Используйте **Guard Clauses** (ранний возврат).

*⛔ Плохо (глубокая вложенность):*
```javascript
function processUser(user) {
  if (user) {
    if (user.isActive) {
      if (user.hasPermission) {
        doSomething(user);
      }
    }
  }
}
```

*✅ Хорошо (плоская структура с ранним возвратом):*
```javascript
function processUser(user) {
  if (!user) return;
  if (!user.isActive) return;
  if (!user.hasPermission) return;
  
  doSomething(user);
}
```

### 2. Избавление от избыточности (Logical De-cluttering)
* Заменяйте длинные ветвления `if-else` на тернарные операторы, логическое ИЛИ/И (`||`, `&&`) или оператор нулевого слияния (`??`), если это улучшает читаемость.
* Удаляйте мертвый код, неиспользуемые переменные, временные логи отладки и забытые блоки комментариев.

### 3. Дедупликация в Splash Radius
* При написании хелперов или утилит проверяйте, не были ли подобные функции уже реализованы в зоне видимости задачи (или в папке `shared/utils/`).
* Избегайте создания локальных копий стандартных процедур (парсинг дат, форматирование валют, обработка ошибок).

## 🔄 Жизненный цикл упрощения в конвейере

1. **Фаза Матроса (Локально)**: Перед отправкой кода Капитану, матрос анализирует свой splash-radius и применяет правила `simplify-skill` к новым файлам.
2. **Фаза Капитана (Глобально - LTO)**: Капитан при сборке проверяет всю совокупность изменений. Для измененных старых файлов генерируется Side-by-Side дифф для оператора.
3. **Авто-снапшот**: Перед записью упрощенного кода на диск всегда выполняется резервное копирование.
4. **Тест-гейт**: Запускается `npm run test`. Если тесты ломаются — рефакторинг автоматически откатывается.
